# gh-monit Roadmap

Product improvement ideas and feature backlog, generated by architecture analysis.

---

## Quick Wins (S effort)

### 1. Bulk Refresh Failure Surfacing
**Problem:** `POST /api/repos/refresh-all` returns `results: { repo: string; success: boolean }[]` but the frontend discards it. Silent per-repo failures during the most common maintenance action undermine trust in the data.

**Fix:** Capture failed repos in `handleRefreshAll()` in `useDashboard.ts` and surface them in a post-refresh banner. ~10 lines of frontend change.

---

### 2. Configurable SLA Thresholds
**Problem:** SLA limits (critical:2d, high:7d, medium:30d, low:90d) are hardcoded. Teams with different policies cannot adapt them without editing source code.

**Fix:** Add a `settings` table (migration v4), `GET /PUT /api/settings` routes, and number inputs in the Settings panel. `getSlaViolations()` already accepts an optional `slaConfig` param — just needs to read from the DB instead of the constant.

---

### 3. Org-Wide Alert Export
**Problem:** Export covers repo summaries or a single repo's alerts. There is no way to export all open alerts across all repos for audits or ticket triage.

**Fix:** Add `GET /api/export/alerts` backed by a `SELECT * FROM alerts WHERE state = 'open'` join. Add "All Alerts (JSON)" and "All Alerts (CSV)" to `ExportMenu`.

---

### 4. Repo-Scoped Analytics
**Problem:** The Analytics tab is org-wide only. There is no way to see a single repo's trend history or MTTR from the UI, even though the API already supports `?repo=` on `/api/history/trends` and `/api/history/mttr`.

**Fix:** Add a repo filter dropdown to the Analytics tab. Zero backend changes needed — pure frontend wiring of existing query params.

---

### 5. Trend Time-Range Selector
**Problem:** The trend chart shows all historical data. For long-running installations this is noisy. Teams want to ask "how have we done in the last 30 days?"

**Fix:** Add `?days=N` support to `/api/history/trends`. Add a 7d / 30d / 90d / All time button group in `TrendChart.tsx`. ~30 lines across 4 files.

---

### 6. Per-Repo Risk Score
**Problem:** Sorting by alert count is crude. Two repos with identical counts can have very different urgency (CVSS scores, SLA overrun).

**Fix:** Compute `risk_score = SUM(cvss_score * severity_weight * overdue_multiplier)` per repo in a new `getRiskScores()` DB function. Add `riskScore` to `RepoSummary`. Add "Sort by Risk" to the repo sort options.

---

## Core Features (M effort)

### 7. Dismissed / Fixed Alert History View
**Problem:** `alert_history` stores the full state transition log but the UI only shows open alerts. Teams doing security reviews need to audit what was dismissed and when.

**Fix:** Add a state filter (open / dismissed / fixed / all) to `AlertsTable`. Extend `GET /api/repos/:owner/:name/alerts` with a `?state=` param. Extract `dismissed_reason` from `raw_json` via `json_extract()`.

---

### 8. Notification Dispatch
**Problem:** A monitoring tool that requires the user to actively open a browser provides minimal security value. Teams cannot reasonably check the dashboard daily across 30+ repos.

**Fix:** After each scheduler run, diff alert counts vs. the previous sync. If net-new critical/high alerts exist, dispatch to:
- **Slack:** `fetch()` to a `GH_MONIT_SLACK_WEBHOOK_URL` env var.
- **Desktop:** `node-notifier` (zero-config, no server dep).

Store opt-in config in the `settings` table (from suggestion 2).

---

### 9. Test Suite Foundation
**Problem:** `npm test` echoes "No tests yet". MTTR computation, SLA violation detection, alert normalization, and the `saveAlerts()` transaction are all business-critical and unprotected.

**Fix:** Add `vitest`. Use `better-sqlite3` in-memory mode (`:memory:`) — the migration system already handles schema creation, so no mocking needed. Priority test targets:
- `normalizeAlerts()` with fixture payloads
- `getSlaViolations()` with seeded alert history
- `getMttrMetrics()` CTE output
- `saveAlerts()` — verify `alert_history` only appends on state changes

---

## Strategic Bets (L effort)

### 10. GitHub Webhook Receiver
**Problem:** The polling model (hourly by default) means new critical alerts can sit unseen for up to 59 minutes. GitHub fires `repository_vulnerability_alert` webhooks in real time.

**Fix:** Add a `gh-monit webhook` CLI command that starts an HTTP listener. Add `POST /webhook` to the Hono server, validating `X-Hub-Signature-256` using `GITHUB_WEBHOOK_SECRET`. Normalize the payload with the existing `normalizeAlerts()` and call `saveAlerts()`. Coexists with the scheduler for repos not covered by the webhook.

## Technical Debt

- **Migration array ordering:** Version 3 is declared before version 2 in the `migrations` array in `db.ts`. `runMigrations` sorts correctly so it works, but it's a readability trap for future contributors. Reorder to be sequential.
- **Cron next-run computation:** `getNextRunDate()` in `scheduler.ts` manually reimplements cron parsing and only handles simple `HH MM * * *` patterns. Complex expressions silently return wrong next-run times in the dashboard UI.
- **Bulk refresh stagger [fixed ✅]:** The 2-second stagger between repo refreshes in `refreshAllRepos` is fixed regardless of repo count. On 50+ repos this is a 100-second blocking request with no streaming or progress signal.
